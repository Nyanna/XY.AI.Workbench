Kurze Antwort: **Ja, das stimmt im Kern** – der live mitlaufende Zeit-/Token-Zähler, den man in der interaktiven TUI (Terminal-UI) sieht (der Spinner mit "⚒ 1.234 tokens · 12s"), ist reine UI-Anzeige und wird im Headless-Modus (`-p` / `--print`) **nicht** live durchgereicht. Aber es gibt trotzdem Wege, an vergleichbare Daten zu kommen – nur eben strukturiert statt als Live-Widget.

**Was man im Headless-Modus tatsächlich bekommt:**

1. **`--output-format json` (Standard bei `-p`):** Am Ende des Laufs liefert Claude Code eine einzelne JSON-Antwort mit Zusammenfassung, u. a.:
```json
{
  "type": "result",
  "total_cost_usd": 0.352,
  "duration_ms": 45000,
  "duration_api_ms": ...,
  "num_turns": 5,
  "usage": {
    "input_tokens": 51,
    "output_tokens": 5555,
    "cache_read_input_tokens": 147615,
    "cache_creation_input_tokens": 12190
  }
}
```
Das entspricht dem, was der Live-Counter am Ende auch anzeigen würde – nur eben erst nach Abschluss, nicht live während der Verarbeitung.

2. **`--output-format stream-json`:** Hier bekommt man newline-delimited JSON-Events (`system/init`, `assistant`, `user`, `result`) während der Verarbeitung. Mit `--include-partial-messages` gibt es sogar granulare Streaming-Deltas. Theoretisch könnte man daraus selbst einen Live-Zähler bauen. **Aber Vorsicht:** Es gibt dokumentierte Bugs dabei:
   - Token-Usage wird bei mehreren Content-Blocks (thinking + text + tool_use) in einer Nachricht **mehrfach gezählt** statt einmal (Issue #6805 – bis zu 8x Inflation).
   - In der Session-JSONL werden teils nur **Zwischenwerte** der `output_tokens` aus Streaming-Deltas gespeichert statt des finalen kumulierten Werts (Issues #21971, #22686) – das führt zu Unterzählung.

3. **Kein eingebautes "Tokens-bis-Kontextlimit"-Signal:** Auch im Headless-SDK fehlt bislang eine klare Angabe zum Kontextfenster-Maximum bzw. Prozentsatz (Feature-Request #8011 ist noch offen).

**Praktische Empfehlung für deinen Anwendungsfall:**
- Für einfache Kostenkontrolle: einfach das finale `result`-Objekt aus `--output-format json` (oder das letzte `result`-Event bei `stream-json`) auswerten – das ist zuverlässig für Gesamtkosten/-tokens am Ende eines Laufs.
- Für **echtes Live-Monitoring** während der Verarbeitung (z. B. in einem eigenen Dashboard) ist die offizielle, robuste Lösung der **OpenTelemetry-Export** von Claude Code, der Token-/Kosten-Metriken nahezu in Echtzeit an ein eigenes Observability-Backend streamt – unabhängig vom stdout-Parsing und ohne die genannten Zähl-Bugs.
- Vom manuellen Aufsummieren der `stream-json`-Deltas würde ich für belastbare Zahlen eher abraten, solange die genannten Duplizierungs-/Untererfassungs-Bugs offen sind.
